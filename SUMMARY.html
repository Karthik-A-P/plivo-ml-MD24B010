Run-by-run reasoning

Run A — Fix the optimizer before anything elsebpb 2.7049

Hypothesis — The starter trainer used constant-LR Adam with no warmup, no weight decay, no gradient clipping, and unscaled initialization. These are cheap, well-understood fixes that should help regardless of tokenizer.

Change — Switched to AdamW (weight decay 0.1 on 2D+ params only), added linear warmup (100 steps) + cosine decay to 10% of peak LR, added gradient-norm clipping at 1.0, tied input/output embeddings, and rescaled residual-branch init by 1/√(2·n_layer) following GPT-2 convention.

Conclusion — Establishes a clean, stable baseline (loss curve fell smoothly 5.51 → 1.93 in the raw loss sense) to isolate the effect of later changes against.

Run C — Try a small BPE vocabbpb 2.2616

Hypothesis — Byte-level tokenization costs 2–3 tokens per Devanagari (Hindi) character, so the 128-token context window covers roughly a third as much real Hindi text as English text. A trained BPE tokenizer should compress this and let each step cover more real content.

Change — Replaced ByteTokenizer with a from-scratch BPE implementation trained only on train_corpus.txt (word-frequency greedy pair merging), target vocab 700.

Conclusion — Confirmed the tokenizer hypothesis directionally (2.7049 → 2.2616), and cut the same dev_eval.txt file from 159,225 byte-tokens down to 64,396 BPE-tokens — a 2.5× compression. But left a question open: was 700 the right vocab size?

Run B — Scale the BPE vocab upbpb 2.1895

Hypothesis — A larger vocab compresses further; is convergence difficulty (more output classes) or context coverage (fewer tokens per window) the dominant factor within a fixed 2,000-step budget?

Change — Retrained BPE to vocab 1500 (dev_eval.txt: 55,641 tokens, 2.9× compression). Same optimizer/init as Run A.

Conclusion — Context coverage won: vocab 1500 beat vocab 700 (2.1895 vs 2.2616) despite having more embedding rows to learn, at a cost of ~128K extra parameters (well inside the 2M cap).

Run D — Swap the optimizer, not just tune itbpb 2.1299 · final

Hypothesis — Community results on fixed-step/fixed-architecture "speedrun" benchmarks report Muon (momentum orthogonalized via Newton-Schulz iteration) reaching a target loss in meaningfully fewer steps than AdamW on 2D transformer weight matrices, by equalizing gradient singular values that AdamW under-weights. This is exactly the fixed-step constraint this assignment imposes.

Change — Hybrid optimizer: Muon (momentum 0.95, 5-step Newton-Schulz, aspect-ratio-scaled updates) for all 2D attention/MLP weight matrices; AdamW retained for embeddings, layer norms, and biases. Same warmup/cosine LR shape, rebased to Muon's LR scale. Tested against the smaller vocab=700 tokenizer to isolate the optimizer's effect cleanly from Run B's vocab change.

Debugging note — First attempt silently trained for only one real step due to an indentation error that moved the forward/backward/step logic outside the training loop (caught because runtime dropped from ~400s to ~1s and bpb landed at the theoretical random-guess value for vocab 700, ln(700)/ln(2) ≈ 9.45 bits/token). Fixed and rerun cleanly.

Conclusion — Muon beat AdamW at the same vocab (2.2616 → 2.1299, a 5.8% relative improvement) using fewer parameters than Run B's AdamW/vocab-1500 result, while still visibly descending in loss at step 2000 — suggesting the optimizer swap had not yet fully converged and may have further headroom beyond this step budget.

Run E — Does Muon still benefit from a bigger vocab?bpb 2.1333

Hypothesis — Under AdamW, scaling vocab 700→1500 clearly helped (Run C → Run B). If that effect is independent of optimizer choice, the same jump should help under Muon too.

Change — Identical Muon+AdamW hybrid setup as Run D, retrained against the vocab=1500 BPE tokenizer instead of vocab=700.

Conclusion — Hypothesis rejected. bpb was flat-to-slightly-worse (2.1299 → 2.1333) despite 128K additional parameters. Muon's per-step optimization efficiency appears to substitute for some of what extra vocab capacity was buying AdamW — so the two levers are not simply additive. Run D (vocab 700) is kept as the final checkpoint: best score, fewest parameters.
